MODULE 7 — Page /history — Historique des commandes passées
================================================================================
  Fichiers : order_success_test.rb · order_close_open_fail_test.rb

7.1  Accès à la page /history
     Avec client@restoqr.ca : password123
     - Accéder à /history
     - Seules les commandes fermées (ended_at non null) sont affichées
     - Chaque commande affiche : table, date, total, lignes, serveur, vibe

7.2  Page vide
     - Si aucune commande fermée → message d'absence d'historique

7.3  Recherche plein texte
     - Champ de recherche disponible (debounce 350ms)
     - Filtre sur : numéro de table, nom d'item, nom du serveur,
       nom du vibe, note, total, date
     - Exemples : « 3 » / « table 3 » → table 3
                  « Tartare » → commandes contenant cet item

7.4  Tri des commandes
     - Sélecteur : « Plus récent » / « Total croissant » / « Total décroissant »
     - Changer le tri relance le chargement

7.5  Filtre par total (sliders min/max)
     - Deux curseurs min et max pour filtrer par montant
     - Le max est calculé dynamiquement selon les commandes du client
     - Saisie directe aussi possible

7.6  Persistance des filtres dans l'URL
     - Les filtres (recherche, tri, min, max) sont synchronisés
       dans les query params → partager l'URL restitue les mêmes filtres

7.7  Évaluation d'un item depuis /history
     - Cliquer sur « Évaluer » sur une ligne → dialogue étoiles + commentaire
     - Appel API : POST /api/reviews
     - L'évaluation apparaît sur la ligne
     - Update via « Modifier », suppression via « Supprimer »

7.8  Évaluation du serveur depuis /history
     - Bouton « Évaluer le serveur » en bas de la carte de commande
     - Même flux que pour un item

8.1  Accès à la page /server
     Avec server@restoqr.ca : password123
     - Accéder à /server
     - Le tableau de bord affiche :
       « Commandes non assignées » et « Mes commandes »

8.2  Accès refusé
     - Sans être connecté → redirection vers /login
     - Avec un compte client → redirection (serverGuard)

8.3  Section « Commandes non assignées »
     - Commandes ouvertes sans serveur assigné
     - Chaque carte affiche : table, personnes, vibe, lignes avec statuts
     - Bouton « Prendre en charge » disponible

8.4  Assigner une commande
     - Cliquer sur « Prendre en charge »
     - Appel API : POST /api/server/orders/:id/assign
     - La commande passe dans « Mes commandes »

8.5  Servir une ligne (ready → served)
     - Dans « Mes commandes », une ligne « Prête » a un bouton « Servir »
     - Cliquer sur « Servir »
     - Appel API : PATCH /api/server/order_lines/:id/serve
     - La ligne passe au statut « Servie »
     - Quand toutes les lignes sont servies, le client voit
       le bouton « Payer » dans /order

8.6  Modifier une ligne (quantité/note)
     - Cliquer sur « Modifier » sur une ligne non servie
     - Appel API : PATCH /api/server/order_lines/:id
     - La ligne est mise à jour

8.7  Supprimer une ligne — hard delete
     - Cliquer sur « Supprimer » (statut sent ou in_preparation)
     - Fenêtre de confirmation
     - Appel API : DELETE /api/server/order_lines/:id
     - La ligne est supprimée définitivement

8.8  Libérer une commande (non payée)
     - Cliquer sur « Libérer la table »
     - Appel API : POST /api/server/orders/:id/release
     - La commande reste dans « Mes commandes » avec un badge « Libérée »
       (server_released = true, ended_at = maintenant)
     - La table reste occupée : le client est déconnecté mais
       la table n'est disponible qu'après « Nettoyer »
     - Aucune nouvelle commande ne peut être créée sur cette table
       tant que le serveur n'a pas nettoyé

8.9  Nettoyer une table (après paiement ou libération)
     - Une commande fermée (ended_at présent) reste visible dans
       « Mes commandes » avec le bouton « Nettoyer la table »
     - Cliquer sur « Nettoyer la table »
     - Appel API : POST /api/server/orders/:id/clean
     - La commande disparaît immédiatement du tableau de bord
     - La table devient disponible (cleaned_at mis à jour)
     - Le nom du serveur reste enregistré dans la commande
       et est visible dans /history

10.3 Créer un vibe
     - Cliquer sur « Ajouter une ambiance »

     - Fenêtre de confirmation

MODULE 11 — Hard delete vs Soft delete
================================================================================
  Fichiers : order_line_destroy_fail_test.rb · server_destroy_line_fail_test.rb
             vibe_destroy_fail_test.rb · vibe_restore_fail_test.rb

11.1  Lignes de commande — Hard delete
      - Depuis /order : DELETE /api/orders/:order_id/order_lines/:id
        (seulement statut « Envoyée »)
      - Depuis /server : DELETE /api/server/order_lines/:id
        (statuts « Envoyée » ou « En préparation »)
      - Depuis /kitchen : DELETE /api/kitchen/order_lines/:id
        (statuts « Envoyée »)
      - Suppression définitive, non récupérable

11.2  Lignes du panier (en attente) — Suppression en mémoire
      - Aucun appel API (la ligne n'est pas encore en base)
      - Perte des items si rechargement de page avant envoi

11.3  Commandes — Fermeture logique (via ended_at)
      - Jamais supprimées physiquement
      - Fermées en remplissant ended_at
      - Restent accessibles dans /history
      - Endpoints : POST /api/orders/:id/pay
                    POST /api/orders/close_open

11.4  Vibes — Soft delete conditionnel (via deleted_at)
      - Sans commandes → hard delete (supprimé physiquement)
      - Avec commandes → soft delete (deleted_at = now)
      - Restauration possible via PATCH /api/vibes/:id/restore
      - Les vibes archivés n'apparaissent pas dans /form

MODULE 12 — Rapport de statistiques — Orders (/kitchen)
================================================================================
  Fichiers : order_success_test.rb (section STATS)
             order_stats_fail_test.rb

  Contexte métier :
    Plusieurs clients peuvent scanner la même table et créer chacun
    leur propre commande, tant que le nombre total de personnes ne
    dépasse pas la capacité de la table (nb_seats). Les commandes
    concurrentes sur une même table héritent de la même vibe et du
    même serveur que la première commande.

12.1 Accès au bouton Statistiques
     Avec admin@restoqr.ca : password123
     - Accéder à /kitchen
     - Un bouton (icône bar_chart) est visible à côté du titre
       « Cuisine » (mat-mini-fab, cuivré, 36×36 px)
     - Ce bouton est invisible pour les cuisiniers et les serveurs
       (contrôle @if authService.isAdmin())

12.2 Accès refusé — non admin
     - Avec cook@restoqr.ca → le bouton n'apparaît pas
     - Avec server@restoqr.ca → le bouton n'apparaît pas
     - Appel direct GET /api/orders/stats sans authentification
       → success: false
     - Appel avec client@restoqr.ca → success: false,
       erreur « Accès réservé aux administrateurs »
     - Appel avec server@restoqr.ca → success: false,
       erreur « Accès réservé aux administrateurs »

12.3 Ouverture du dialogue de statistiques
     - Cliquer sur le bouton bar_chart
     - Le dialogue « Rapport de statistiques — Orders » s'ouvre
     - Un spinner s'affiche pendant le chargement
     - Appel API : GET /api/orders/stats
     - La réponse contient : columns, rows, details

12.4 Colonnes du tableau de statistiques
     Le tableau mat-table affiche les colonnes suivantes,
     groupées par table + vibe + serveur :
       · Table           — numéro de la table
       · Vibe            — nom de l'ambiance
       · Serveur         — prénom + nom du serveur
       · Nb commandes    — nombre de commandes pour ce groupe
       · Nb lignes       — nombre total de lignes de commande
       · Qté totale      — somme des quantités commandées
       · Qté moy.        — moyenne des quantités par commande
       · Revenu ($)      — somme (quantité × prix unitaire)
       · Pourboires ($)  — somme des pourboires
       · Total tips+revenu ($) — revenu + pourboires
       · Prix moy. ligne ($)  — prix moyen par ligne
       · Item #1         — item le plus commandé (par quantité)
       · Combo #1        — combo le plus commandé (par quantité)
       · Durée moy. (min) — durée moyenne commande (created_at → ended_at)

12.5 Filtres disponibles
     - Date début : sélecteur de date (mat-datepicker)
       → recharge les stats avec start_date
     - Date fin : sélecteur de date
       → recharge les stats avec end_date
     - Vibes : sélecteur multiple (mat-select)
       → recharge les stats avec category_ids[]
     - Les trois filtres sont combinables

12.6 Lignes cliquables — drill-down par table
     - Chaque ligne du tableau est cliquable (curseur pointer)
     - Cliquer sur une ligne déplie un panneau de détail
       sous cette ligne uniquement
     - Cliquer une deuxième fois la replie
     - Seule la ligne cliquée se déplie (clé unique :
       table_number + vibe_name + server_name)

12.7 Panneau de détail — commandes de la table
     - Titre : « Commandes — Table X »
     - Liste des commandes filtrées par table + vibe + serveur
     - Chaque commande affiche :
       · Heure de début (formatée : mois jour, HH:MM)
       · Heure de fin (si fermée)
       · Nombre de personnes
       · Nom du serveur
       · Badge vibe
       · Montant (revenu) en $
       · Pourboire (si > 0)
       · Badge « Ouverte » ou « Fermée »
     - Cliquer sur une commande déplie ses lignes de commande

12.8 Sous-panneau — lignes de commande d'une commande
     - Note de la commande (si présente)
     - Tableau avec les colonnes :
       · Article — nom de l'item ou combo
       · Qté — quantité commandée
       · Prix unit. — prix unitaire
       · Total — quantité × prix unitaire
       · Statut — badge coloré (En attente / Envoyé /
                  En préparation / Prêt / Servi)
       · Note — note de la ligne ou « — »

12.9 Bouton Fermer
     - Un bouton « Fermer » en bas à droite du dialogue
     - Ferme le dialogue sans action

12.10 Seeds — commandes concurrentes
      - Table 3 (4 places) contient 2 commandes ouvertes
        simultanées :
        · demo@restoqr.ca — 2 personnes, vibe Fête, serveur Marie
        · client@restoqr.ca — 1 personne, même vibe, même serveur
      - Cela démontre le scénario « ami qui scanne la même table »
      - La capacité totale (2 + 1 = 3) ne dépasse pas les 4 places
